# Access Request Ticket Triage — State Machine Design

For an access issue / access request, the state machine models the ticket as moving through a sequence of decision states based on events.

---

## Core Concepts

| Concept | Definition |
|---------|-----------|
| **State** | The current triage condition of the ticket |
| **Event** | Something that happens and may cause a state transition |
| **Action** | What the agent or human does during or after a transition |

---

## Abstract State Shape

A reusable, machine-readable representation of a ticket:

```json
{
  "bug_id": "string",
  "rulebook": "access_request_v1",
  "state": "string",
  "category": "string",
  "subcategory": "string | null",
  "entities": {},
  "missing_fields": [],
  "flags": {
    "is_duplicate_candidate": false,
    "needs_approval": true,
    "is_blocked": true,
    "requires_human_review": true
  },
  "last_event": "string",
  "allowed_next_events": [],
  "recommended_action": {
    "type": "string",
    "target": "string | null",
    "message": "string"
  },
  "audit": [
    {
      "event": "string",
      "from_state": "string",
      "to_state": "string",
      "reason": "string"
    }
  ]
}
```

### Key Fields

- **`state`** — One value from a predefined enum in the rule book.
- **`last_event`** — Also an enum; events drive transitions.
- **`entities`** — Schema predefined; values inferred from ticket content.
- **`missing_fields`** — Derived from the rule book; agent compares required fields against extracted entities.
- **`recommended_action.type`** — Predefined enum; the specific value at each step is inferred.
- **`audit`** — Schema predefined; reasons and evidence are generated by the agent.

> **Predefined lists control the workflow. Agentic inference fills in evidence and values inside that workflow.**
> That is what makes the system testable, debuggable, and safe.

## Predefined Enums for Access Tickets

### States

| State | Meaning |
|-------|---------|
| `new` | Ticket has just arrived |
| `intake` | Agent is extracting fields and classifying |
| `missing_info` | Required access fields are missing |
| `duplicate_review` | Possible duplicate needs confirmation |
| `ready_for_access_review` | Enough info exists to start access review |
| `approver_review` | Waiting for lead/approver decision |
| `access_provisioning` | Access admin/team needs to grant the role |
| `verification` | Waiting for user to confirm access works |
| `resolved` | Access appears completed |
| `closed` | Ticket is closed |
| `denied` | Approver denied access |
| `stale_waiting_for_user` | User has not responded for too long |
| `human_review` | Needs manual judgment or exception handling |

## Design Principle

> The abstract state should **not** be the full ticket. It should be the minimal machine-readable summary needed to decide the next transition based on a predefined rule book.
